# Dynamic DSL Reconciliation Framework — Canonical Architecture

## 1. Purpose
This document defines the architecture for the dynamic reconciliation
framework that compares DSL (flattened JSON) data against any external
system (Onyx, ArcGIS, WinCan, etc.).

Reconciliation direction is always:

    DSL  →  Target System

DSL is the canonical, BIAN-aligned truth snapshot.
All reconciliations use DSL as the left-hand side.

The architecture is:
- deterministic
- metadata-driven
- system-agnostic
- domain/entity-agnostic
- scalable
- sponsor-grade
- audit-safe

Once all systems reshape their data to the DSL grain, the engine becomes
fully dynamic with no join rules, no polymorphism, and no entity-specific SQL.

---

## 2. Metadata Model (System-Agnostic)

The metadata model supports reconciliation against any system.

Fields:

- domain
- entity
- system_name
- dsl_attribute
- system_attribute
- is_key
- reconciliation_type
- is_active

### Example (bian_mappings.csv)

domain,entity,system_name,dsl_attribute,system_attribute,is_key,reconciliation_type,is_active
Party,Customer,Onyx,partyReferenceId,partyReferenceId,Y,EXACT,Y
Party,Customer,Onyx,fullName,full_name,N,NORMALIZED,Y
Party,Customer,ArcGIS,fullName,asset_label,N,NORMALIZED,Y
Party,Customer,WinCan,fullName,PCDStartStreetName,N,NORMALIZED,Y

This metadata is the single source of truth for reconciliation.

## 3. Schema Strategy (Staging + Reporting)

dbt publishes into predefined schemas:

### staging schema
Contains reshaped, grain-aligned inputs:

- stg_dsl_<domain>_<entity>
- stg_<system_name>_<domain>_<entity>

### reporting schema
Contains reconciliation outputs:

- rec_<system_name>_<domain>_<entity>
- reconciliation_unified

The schema is fixed by dbt.
The alias is dynamic and generated by the reconciliation macro.

This preserves full dynamic reconciliation while respecting schema boundaries.

## 4. System Source Views (Reshaped to DSL Grain)

Each system must reshape its raw data into domain/entity-specific staging views
that expose `partyReferenceId` and align with the DSL grain.

### Example: Onyx

#### Party.Customer
SELECT
    owner_id AS partyReferenceId,
    full_name,
    email_address,
    phone_number
FROM raw_onyx_owner
WHERE owner_type = 'Customer';

#### Party.Address
SELECT
    address_id AS partyReferenceId,
    address_line_1,
    postcode
FROM raw_onyx_addresses
WHERE address_type = 'Physical';

#### Party.Email
SELECT
    email_party_id AS partyReferenceId,
    email_address
FROM raw_onyx_emails;

#### Party.Phone
SELECT
    phone_party_id AS partyReferenceId,
    phone_number
FROM raw_onyx_phones;

Any system can be added by creating equivalent views.

## 5. Join Logic (Universal)

All reconciliations use the same join:

{% macro build_join(domain, entity) %}
    ON dsl.partyReferenceId = system.partyReferenceId
{% endmacro %}

No join rules.
No crm_key_type.
No mapping_rule.
No polymorphism.

## 6. Comparison Logic

Comparison is driven entirely by reconciliation_type:

- EXACT
- NORMALIZED
- DATE_ONLY

{% macro build_comparison_expression(dsl_col, system_col, reconciliation_type) %}
    {% if reconciliation_type == 'EXACT' %}
        {{ dsl_col }} = {{ system_col }}
    {% elif reconciliation_type == 'NORMALIZED' %}
        lower(trim({{ dsl_col }})) = lower(trim({{ system_col }}))
    {% elif reconciliation_type == 'DATE_ONLY' %}
        date({{ dsl_col }}) = date({{ system_col }})
    {% else %}
        {{ dsl_col }} = {{ system_col }}
    {% endif %}
{% endmacro %}

## 7. Dynamic Reconciliation Macro (System-Agnostic)

{% macro dynamic_reconciliation(domain, entity, system_name) %}

{{ config(
    schema='reporting',
    alias='rec_' ~ system_name ~ '_' ~ domain ~ '_' ~ entity
) }}

WITH meta AS (
    SELECT *
    FROM {{ ref('bian_mappings_normalized') }}
    WHERE domain = '{{ domain }}'
      AND entity = '{{ entity }}'
      AND system_name = '{{ system_name }}'
      AND is_active = 'Y'
),

joined AS (
    SELECT
        dsl.*,
        system.*,
        {{ dbt_utils.generate_surrogate_key(
            meta | where("is_key = 'Y'") | map(attribute='dsl_attribute')
        ) }} AS pk_hash
    FROM {{ ref('stg_dsl_' ~ domain ~ '_' ~ entity) }} AS dsl
    LEFT JOIN {{ ref('stg_' ~ system_name ~ '_' ~ domain ~ '_' ~ entity) }} AS system
        {{ build_join(domain, entity) }}
),

compare AS (
    SELECT
        '{{ domain }}' AS domain,
        '{{ entity }}' AS entity,
        '{{ system_name }}' AS system_name,
        pk_hash,
        m.dsl_attribute AS attribute_name,
        m.reconciliation_type,
        CASE
            WHEN {{ build_comparison_expression(
                    'dsl.' ~ m.dsl_attribute,
                    'system.' ~ m.system_attribute,
                    m.reconciliation_type
                 ) }}
            THEN 'MATCH'
            ELSE 'MISMATCH'
        END AS match_status,
        CASE
            WHEN {{ build_comparison_expression(
                    'dsl.' ~ m.dsl_attribute,
                    'system.' ~ m.system_attribute,
                    m.reconciliation_type
                 ) }}
            THEN NULL
            ELSE 'Comparison failed: ' || m.reconciliation_type
        END AS mismatch_reason
    FROM joined
    CROSS JOIN meta AS m
)

SELECT * FROM compare;

{% endmacro %}

## 8. Unified Reconciliation Ledger

The engine writes metadata-only mismatch rows into:

reporting.reconciliation_unified

Fields:

- domain
- entity
- system_name
- primary_key_hash
- attribute_name
- reconciliation_type
- match_status
- mismatch_reason
- run_timestamp

No values are stored.
